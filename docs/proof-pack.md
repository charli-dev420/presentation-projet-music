## English

### Purpose

The proof pack gives outside readers a usable evidence map without publishing raw internals. It is designed for buyer diligence, partner scoping, tester onboarding, recruiter review, and funding conversations.

### Proof Lanes

| Lane | Public proof signal | Confidence use | Good next request |
| --- | --- | --- | --- |
| Product center | DAW Core is consistently positioned as the main product across README, one-pager, flows, and buyer material. | Strong for narrative clarity. | Ask for one focused DAW Core demo. |
| Portable project | `.dw` contract covers state, assets, snapshots, checksums, reload, and re-export expectations. | Strong as a product contract. | Ask to replay the scenario on the target surface. |
| Desktop validation | Desktop complete-gate signal summarized as 25/25 with `findings=0` on 2026-06-23. | Strong as desktop maturity signal. | Ask for the current release-scope summary if release is the decision. |
| Android beta | Android is framed as beta/recertification with gate language. | Useful for planning testers and device matrix. | Ask for a current device-run summary. |
| Synth suite | Seven-instrument UWdeVST family has shared UX and automated technical QA signals. | Strong for suite coherence, pending listening sign-off where relevant. | Ask for listening review and release-candidate criteria. |
| VST distribution | Public asset/manual inventory and showcase images are documented. | Useful for distribution readiness conversation. | Ask for scoped private catalog/installer review. |
| FX suite | Effects are grouped by musical treatment families. | Useful for product strategy. | Ask which FX family most improves DAW Core's next workflow. |

### What The Pack Deliberately Excludes

The public proof pack excludes raw logs, APKs, VST binaries, installers, private presets, CSV QA dumps, device identifiers, user sessions, local paths, private storage details, secrets, and source code.

That exclusion is part of the quality posture: public proof should be readable and safe; private proof should be shared only when the review scope justifies it.

### Partner Proof Request Template

Use this format when asking for a private proof replay:

| Field | Fill in |
| --- | --- |
| Surface | DAW Core desktop, Android beta, synth suite, FX suite, VST distribution. |
| Scenario | The exact workflow to demonstrate. |
| Success criterion | What must be true for the review to pass. |
| Evidence level | Public summary, redacted artifact, NDA artifact, live demo, source walkthrough. |
| Privacy boundary | What must not leave the private environment. |
| Follow-up | Fix, retest, partner decision, funding decision, hiring/mission discussion. |

### Ce que le pack exclut volontairement

Le proof pack public exclut logs bruts, APKs, binaires VST, installateurs, presets privés, dumps CSV QA, identifiants device, sessions utilisateur, chemins locaux, détails storage privés, secrets et code source.

Cette exclusion fait partie de la posture qualité: la preuve publique doit être lisible et sûre; la preuve privée doit être partagée seulement quand le périmètre de review le justifie.

### Template de demande preuve partenaire

Utiliser ce format pour demander un replay privé:

| Champ | A renseigner |
| --- | --- |
| Surface | DAW Core desktop, beta Android, suite synthés, suite FX, distribution VST. |
| Scénario | Workflow exact à démontrer. |
| Critère succès | Ce qui doit être vrai pour passer la review. |
| Niveau de preuve | Synthèse publique, artefact redacted, artefact NDA, démo live, walkthrough source. |
| Limite privée | Ce qui ne doit pas sortir de l'environnement privé. |
| Suite | Fix, retest, décision partenaire, financement, discussion poste/mission. |
