<h2 id="english">🇬🇧 English</h2>

This page gives a short review path for someone discovering the public music repository.

The goal is to understand the project quickly without reading every document in detail.

---

## Five-minute path

Recommended reading order:

| Step | Page | What to understand |
| --- | --- | --- |
| 1 | `README.md` | The public entry point and the split between DAWWW-CORE and UnicorSoundEngine. |
| 2 | `docs/project-overview.md` | The main DAWWW-CORE product story. |
| 3 | `docs/dawww-core.md` | The DAW surfaces, `.dw` workflow and platform scope. |
| 4 | `docs/unicorsoundengine.md` | The separate plugin and audio-product line. |
| 5 | `docs/proof-summary.md` | What can be said publicly and what remains controlled. |
| 6 | `docs/open-needs.md` | Useful review, feedback and collaboration areas. |

---

## What should be clear after five minutes

After this short review, a reader should understand that:

- DAWWW-CORE is the main browser DAW / local-first music workspace;
- UnicorSoundEngine is a separate VST / plugin ecosystem;
- DAWWW-CORE does not load native VST plugins in the browser;
- the `.dw` format is central to the DAWWW-CORE continuity story;
- the public repository is a showcase and documentation layer, not the full private project archive;
- detailed proof, private QA, transfer notes and handover material belong to controlled review.

---

## What to check first

A useful first review should focus on clarity:

- is the separation between DAWWW-CORE and UnicorSoundEngine understandable;
- does the README stay short enough;
- are the public claims careful enough;
- does the `.dw` project story make sense;
- is Android described as a direction without overclaiming;
- are the public proof limits clear.

---

## Good review output

A useful external review can be short.

The most helpful format is:

```text
What I read:
What I understood:
What was unclear:
What sounded too strong:
What proof I would expect next:
```

This keeps feedback concrete and avoids vague comments.

<h2 id="francais">🇫🇷 Français</h2>

Cette page donne un parcours de revue rapide pour quelqu’un qui découvre le dépôt music public.

Le but est de comprendre le projet rapidement sans lire tous les documents en détail.

---

## Parcours en cinq minutes

Ordre de lecture recommandé :

| Étape | Page | Ce qu’il faut comprendre |
| --- | --- | --- |
| 1 | `README.md` | Le point d’entrée public et la séparation entre DAWWW-CORE et UnicorSoundEngine. |
| 2 | `docs/project-overview.md` | Le récit produit principal de DAWWW-CORE. |
| 3 | `docs/dawww-core.md` | Les surfaces DAW, le workflow `.dw` et le périmètre plateforme. |
| 4 | `docs/unicorsoundengine.md` | La ligne plugin et produit audio séparée. |
| 5 | `docs/proof-summary.md` | Ce qui peut être dit publiquement et ce qui reste contrôlé. |
| 6 | `docs/open-needs.md` | Les retours, revues et collaborations utiles. |

---

## Ce qui doit être clair après cinq minutes

Après cette revue courte, un lecteur doit comprendre que :

- DAWWW-CORE est le DAW navigateur / workspace musical local-first principal ;
- UnicorSoundEngine est un écosystème VST / plugin séparé ;
- DAWWW-CORE ne charge pas de VST natifs dans le navigateur ;
- le format `.dw` est central dans le récit de continuité DAWWW-CORE ;
- le dépôt public est une vitrine et une couche documentaire, pas l’archive privée complète du projet ;
- les preuves détaillées, la QA privée, les notes de transfert et le handover appartiennent à une revue contrôlée.

---

## Ce qu’il faut vérifier en premier

Une première revue utile doit se concentrer sur la clarté :

- la séparation entre DAWWW-CORE et UnicorSoundEngine est-elle compréhensible ;
- le README reste-t-il assez court ;
- les claims publics sont-ils assez prudents ;
- le récit projet `.dw` est-il clair ;
- Android est-il décrit comme une direction sans surpromesse ;
- les limites de preuve publique sont-elles claires.

---

## Format de retour utile

Une revue externe utile peut être courte.

Le format le plus efficace est :

```text
Ce que j’ai lu :
Ce que j’ai compris :
Ce qui reste flou :
Ce qui semble trop fort :
La preuve que j’attendrais ensuite :
```

Cela garde le feedback concret et évite les remarques vagues.
